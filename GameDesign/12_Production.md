# 12. La production d'un jeu vidéo

## Objectifs pédagogiques

À la fin de ce chapitre, vous serez capable de :

- Comprendre les différentes phases de production d'un jeu vidéo.
- Identifier les rôles au sein d'une équipe de développement.
- Utiliser les méthodes de gestion de projet les plus courantes.
- Organiser efficacement le travail d'une équipe.
- Gérer les risques d'un projet.
- Comprendre le cycle de vie complet d'un jeu, de l'idée à sa sortie.
- Préparer une production réaliste, quel que soit la taille du studio.

---

# 12.1 Qu'est-ce que la production ?

La **production** regroupe toutes les activités permettant de transformer une idée en jeu commercialisable.

Contrairement au Game Design, la production ne définit pas **ce qu'est le jeu**, mais **comment il sera réalisé**.

Le producteur veille notamment à :

- respecter les délais ;
- gérer le budget ;
- coordonner les équipes ;
- anticiper les risques ;
- maintenir une vision cohérente du projet.

Sans production, même un excellent concept peut échouer.

---

# 12.2 Les grandes phases d'un projet

Le développement d'un jeu suit généralement plusieurs étapes.

## 1. Concept

L'équipe définit :

- l'idée générale ;
- le public cible ;
- les plateformes ;
- les objectifs.

À ce stade, tout peut encore évoluer.

---

## 2. Pré-production

La pré-production sert à répondre à une question essentielle :

> **"Sommes-nous capables de fabriquer ce jeu ?"**

On réalise notamment :

- prototypes ;
- premiers Game Design Documents ;
- direction artistique ;
- architecture technique ;
- estimation des coûts ;
- premiers plannings.

La plupart des décisions importantes sont prises ici.

---

## 3. Production

La production représente la plus grande partie du projet.

Les équipes créent :

- niveaux ;
- personnages ;
- musiques ;
- interfaces ;
- systèmes ;
- quêtes ;
- cinématiques.

Les fonctionnalités deviennent progressivement complètes.

---

## 4. Alpha

Toutes les fonctionnalités prévues sont présentes.

Le contenu principal est terminé.

Cependant :

- bugs ;
- équilibrage ;
- optimisation.

restent à traiter.

On dit souvent :

> **Feature Complete**

---

## 5. Bêta

Le jeu est jouable du début à la fin.

Les nouvelles fonctionnalités sont limitées.

L'objectif devient :

- corriger ;
- optimiser ;
- équilibrer.

---

## 6. Gold Master

Version envoyée :

- aux constructeurs ;
- aux plateformes ;
- aux éditeurs.

Cette version sert de référence pour la sortie.

---

## 7. Live Ops

Pour de nombreux jeux modernes, le développement continue après la sortie.

Exemples :

- nouvelles saisons ;
- événements ;
- équilibrages ;
- DLC ;
- correctifs.

Le jeu devient un service.

---

---

Il ne décide pas forcément du Game Design.

Son rôle consiste à garantir que le projet avance.

# 12.4 Les méthodes Agile

Aujourd'hui, la majorité des studios utilisent des méthodes dites **Agiles**.

Le principe :

Produire rapidement.

Tester souvent.

Corriger régulièrement.

Au lieu d'attendre plusieurs années avant de découvrir les problèmes.

---

# 12.5 Scrum

Scrum est l'une des méthodes Agile les plus utilisées.

Elle repose sur plusieurs éléments.

## Sprint

Période de travail courte.

Souvent :

1 à 3 semaines.

Chaque Sprint possède un objectif clair.

---

## Backlog

Liste priorisée des tâches.

Exemples :

- créer un ennemi ;
- corriger un bug ;
- équilibrer une arme.

---

## Daily Meeting

Réunion quotidienne très courte.

Chaque membre répond généralement à trois questions :

- Qu'ai-je fait hier ?
- Que vais-je faire aujourd'hui ?
- Quels sont mes blocages ?

---

## Sprint Review

Présentation du travail terminé.

---

## Rétrospective

L'équipe analyse son fonctionnement.

Objectif :

Améliorer la méthode de travail.

---

# 12.6 Les jalons (Milestones)

Un projet est découpé en objectifs intermédiaires.

Exemple :

Prototype

↓

Vertical Slice

↓

Alpha

↓

Bêta

↓

Gold

Chaque jalon permet de mesurer les progrès.

---

# 12.7 La gestion des risques

Tous les projets comportent des risques.

Quelques exemples :

- retard ;
- départ d'un développeur ;
- technologie instable ;
- dépassement du budget ;
- mécanique qui ne fonctionne pas.

Le rôle du Producteur est d'anticiper ces situations.

---

# 12.8 Les priorités

Toutes les fonctionnalités n'ont pas la même importance.

Une méthode classique consiste à distinguer :

## Must Have

Indispensable.

Sans cette fonctionnalité, le jeu ne fonctionne pas.

---

## Should Have

Très importante.

Peut être reportée si nécessaire.

---

## Could Have

Amélioration souhaitable.

---

## Won't Have

Écartée de cette version.

Peut être intégrée plus tard.

Cette méthode est connue sous le nom de **MoSCoW**.

---

# 12.9 Le Scope

Le **Scope** représente l'ensemble des fonctionnalités du projet.

Le principal danger est le **Scope Creep** :

Ajouter continuellement de nouvelles idées.

Conséquences :

- retards ;
- budget dépassé ;
- équipe démotivée.

Un bon Producteur protège le projet contre cette dérive.

---

# 12.10 Les indicateurs

Pour suivre un projet, plusieurs indicateurs sont utilisés.

Exemples :

- tâches terminées ;
- bugs ouverts ;
- bugs corrigés ;
- vélocité ;
- temps restant ;
- avancement des fonctionnalités.

Les indicateurs servent à prendre des décisions, pas à surveiller les individus.

# 12.12 Les réunions

Toutes les réunions doivent avoir :

- un objectif ;
- une durée limitée ;
- une liste de décisions.

Une réunion sans décision est souvent une perte de temps.

---

# 12.13 Les tests qualité

Le QA ne consiste pas uniquement à trouver des bugs.

Il vérifie également :

- équilibrage ;
- progression ;
- performances ;
- accessibilité ;
- stabilité.

Chaque bug est généralement documenté.

Exemple :

Titre

Description

Étapes de reproduction

Résultat attendu

Résultat obtenu

Priorité

---

# 12.14 Après la sortie

Le travail ne s'arrête pas.

Les équipes analysent :

- statistiques ;
- avis ;
- ventes ;
- crashs ;
- comportement des joueurs.

Ces données servent à préparer :

- correctifs ;
- extensions ;
- futures mises à jour.

---

# 12.15 Étude de cas : Stardew Valley

Développé presque entièrement par une seule personne.

Le projet montre qu'une excellente organisation peut compenser une petite équipe.

Le scope est resté maîtrisé.

Les mises à jour ont été publiées progressivement après la sortie.

---

# 12.16 Étude de cas : Fortnite

Le développement continue en permanence.

Chaque saison apporte :

- nouveaux contenus ;
- équilibrages ;
- événements ;
- mécaniques.

Le jeu fonctionne selon un modèle de **Live Service**.

La production ne s'arrête jamais réellement.

---

# 12.17 Les erreurs fréquentes

## Vouloir tout faire

Un projet réaliste vaut mieux qu'un projet gigantesque jamais terminé.

---

## Négliger la pré-production

Les erreurs découvertes tardivement coûtent très cher.

---

## Ajouter constamment des fonctionnalités

Le Scope Creep est l'une des principales causes d'échec.

---

## Absence de documentation

Les équipes perdent du temps à rechercher des informations.

---

## Ne pas tester

Les bugs et les problèmes de gameplay s'accumulent.

# Conclusion de la formation

Le Game Design est une discipline à la croisée de nombreux domaines :

- psychologie ;
- ergonomie ;
- narration ;
- programmation ;
- level design ;
- économie ;
- production.

Le rôle du Game Designer n'est pas simplement d'avoir des idées.

Son véritable travail consiste à **concevoir des systèmes capables de produire une expérience mémorable**, puis à les améliorer grâce à l'observation, aux tests et à l'itération.

Les meilleurs jeux ne naissent pas d'une idée géniale.

Ils naissent d'un processus rigoureux où chaque hypothèse est testée, remise en question et perfectionnée.

Comme le résume souvent l'industrie :

> **"Fail faster, learn faster."**

Autrement dit :

Prototypez rapidement.

Testez souvent.

Écoutez les joueurs.

Supprimez ce qui ne fonctionne pas.

Renforcez ce qui procure du plaisir.

Le Game Design est moins l'art d'inventer que celui de sélectionner les bonnes idées, au bon moment, pour les bons joueurs.

# À retenir

- Un jeu est avant tout une expérience interactive.
- Le Game Designer conçoit des règles, pas uniquement des idées.
- Les mécaniques, systèmes, niveaux, interfaces et récits doivent former un ensemble cohérent.
- Les prototypes et les playtests sont les outils les plus puissants du concepteur.
- Une bonne production repose sur une vision claire, un scope maîtrisé et une itération constante.
- La réussite d'un projet dépend autant de sa conception que de son exécution.
