# 10. Le prototypage et les Playtests

## Objectifs pédagogiques

À la fin de ce chapitre, vous serez capable de :

- Comprendre pourquoi le prototypage est indispensable.
- Choisir le bon type de prototype selon vos objectifs.
- Concevoir un prototype rapidement.
- Organiser un playtest efficace.
- Observer les joueurs sans les influencer.
- Transformer les retours en améliorations concrètes.
- Itérer jusqu'à obtenir un gameplay satisfaisant.

---

# 10.1 Pourquoi prototyper ?

L'une des erreurs les plus fréquentes chez les débutants est de vouloir créer directement le jeu final.

Ils commencent par :

- les graphismes ;
- les animations ;
- le scénario ;
- les effets spéciaux.

Puis découvrent plusieurs mois plus tard que...

...le jeu n'est pas amusant.

Le prototype sert précisément à éviter cette situation.

Il permet de répondre rapidement à une question essentielle :

> **"Cette idée est-elle amusante ?"**

---

# 10.2 Qu'est-ce qu'un prototype ?

Un prototype est une version simplifiée d'un jeu.

Son objectif n'est pas d'être beau.

Son objectif est de tester une hypothèse.

Exemples :

- Le saut est-il agréable ?
- Les combats sont-ils lisibles ?
- La boucle de gameplay est-elle intéressante ?
- Les joueurs comprennent-ils les objectifs ?

Le prototype répond à une seule question à la fois.

---

# 10.3 Les différents types de prototypes

## Prototype papier (Paper Prototype)

Le plus rapide.

Utilisé pour tester :

- règles ;
- économie ;
- cartes ;
- interfaces ;
- gameplay tour par tour.

Matériel :

- papier ;
- crayons ;
- post-it ;
- dés ;
- cartes.

Coût :

Très faible.

Temps :

Quelques heures.

---

## Greybox

Le gameplay est testé avec des formes simples.

Exemples :

- cubes ;
- capsules ;
- cylindres.

Les graphismes sont volontairement absents.

Le but est de tester :

- déplacements ;
- sauts ;
- niveaux ;
- visibilité.

---

## Prototype fonctionnel

Les mécaniques principales fonctionnent.

Le contenu est limité.

Il sert à vérifier :

- le Game Feel ;
- les contrôles ;
- les combats ;
- les interactions.

---

## Vertical Slice

Il ne s'agit plus d'un prototype.

Le Vertical Slice représente une petite partie du jeu final.

Tout y est présent :

- graphismes ;
- sons ;
- interface ;
- gameplay ;
- narration.

Il permet notamment de convaincre un éditeur ou un investisseur.

---

# 10.4 Le principe du MVP

Dans le développement de jeux, on applique souvent le concept de **Minimum Viable Product (MVP)**.

L'idée est simple :

Créer la plus petite version possible du jeu.

Exemple :

Avant de créer 100 armes...

Créer une seule arme.

Avant de créer 50 niveaux...

Créer un seul niveau.

Avant d'écrire 10 heures de scénario...

Créer une seule quête.

---

# 10.5 Le cycle de prototypage

Le travail du Game Designer suit généralement une boucle.

Idée

↓

Prototype

↓

Test

↓

Analyse

↓

Modification

↓

Nouveau prototype

Cette boucle peut être répétée plusieurs dizaines de fois.

Le jeu s'améliore progressivement.

---

# 10.6 Les hypothèses

Chaque prototype doit tester une hypothèse.

Exemple :

Hypothèse :

> "Le double saut rendra l'exploration plus amusante."

Le prototype sert uniquement à vérifier cette idée.

À la fin du test :

- hypothèse validée ;
- hypothèse rejetée ;
- hypothèse à modifier.

---

# 10.7 Les Playtests

Le Playtest consiste à observer une personne jouant au prototype.

Le Game Designer ne teste pas son propre jeu.

Il observe d'autres joueurs.

Pourquoi ?

Parce que le concepteur connaît déjà toutes les règles.

Le joueur, lui, les découvre.

---

# 10.8 Observer sans intervenir

Pendant un Playtest :

Ne pas expliquer.

Ne pas guider.

Ne pas aider.

L'objectif est de voir :

- où le joueur hésite ;
- où il se trompe ;
- où il s'ennuie ;
- où il s'amuse.

Chaque difficulté observée est une information précieuse.

---

# 10.9 Les données qualitatives

Les observations sont souvent plus utiles que les opinions.

Observer :

- les hésitations ;
- les sourires ;
- les soupirs ;
- les abandons ;
- les moments de concentration.

Ces réactions révèlent la qualité de l'expérience.

---

# 10.10 Les données quantitatives

Les chiffres permettent de confirmer les observations.

Exemples :

- durée d'une partie ;
- nombre de morts ;
- nombre de tentatives ;
- objets utilisés ;
- objets ignorés ;
- temps passé dans chaque zone.

Les grands studios analysent ces données automatiquement.

---

# 10.11 Les bonnes questions

Après un Playtest, évitez les questions vagues.

Mauvaise question :

> "Tu as aimé ?"

Meilleures questions :

- Quel moment as-tu préféré ?
- À quel moment étais-tu perdu ?
- Quelle mécanique t'a semblé inutile ?
- Quel passage était frustrant ?
- As-tu compris ton objectif ?

Ces réponses sont exploitables.

---

# 10.12 Les faux problèmes

Les joueurs identifient très bien les problèmes.

En revanche...

Ils proposent rarement les bonnes solutions.

Exemple :

Un joueur dit :

> "Il faudrait ajouter un double saut."

En réalité, le vrai problème est peut-être :

Le niveau est trop grand.

Le rôle du Game Designer consiste à comprendre la cause réelle.

---

# 10.13 Prioriser les retours

Tous les retours ne se valent pas.

Une méthode simple consiste à classer les problèmes.

## Critique

Empêche de jouer.

Exemple :

Le joueur reste bloqué.

---

## Majeur

Le gameplay fonctionne mal.

Exemple :

Une mécanique est incomprise.

---

## Mineur

Le jeu reste agréable.

Quelques améliorations sont possibles.

---

## Cosmétique

Petits détails visuels.

À traiter en dernier.

---

# 10.14 Quand arrêter d'itérer ?

Un prototype n'est jamais parfait.

Il faut savoir s'arrêter.

On considère souvent qu'une mécanique est suffisamment mature lorsque :

- les joueurs la comprennent rapidement ;
- ils prennent du plaisir ;
- peu d'explications sont nécessaires ;
- les retours deviennent similaires.

---

# 10.15 Les erreurs fréquentes

## Ajouter du contenu trop tôt

Plus de contenu ne résout pas un gameplay médiocre.

Il vaut mieux améliorer une bonne mécanique que multiplier les fonctionnalités.

---

## Tester avec uniquement ses amis

Les proches hésitent parfois à critiquer.

Il est préférable de tester avec différents profils de joueurs.

---

## Ignorer les observations

Les données recueillies doivent conduire à des actions concrètes.

---

## Modifier plusieurs éléments à la fois

Changer cinq variables simultanément rend difficile l'identification de la véritable cause d'une amélioration ou d'une régression.

Privilégiez des modifications ciblées.

---

# 10.16 Les outils de prototypage

Selon le type de projet, différents outils peuvent être utilisés.

## Papier

- jeux de société ;
- interfaces ;
- systèmes.

---

## Figma

- interfaces ;
- wireframes ;
- menus.

---

## Unity

- jeux 2D ;
- prototypes rapides ;
- gameplay.

---

## Unreal Engine

- jeux 3D ;
- Blueprints ;
- prototypes visuels.

---

## Godot

- projets indépendants ;
- prototypes légers.

---

## Tableurs (Excel, Google Sheets)

Parfaits pour tester :

- économie ;
- progression ;
- statistiques ;
- équilibrage.

---

# 10.17 Étude de cas : Valve

Valve teste continuellement ses jeux.

Les développeurs observent :

- où les joueurs regardent ;
- où ils meurent ;
- où ils hésitent.

De nombreuses décisions de Level Design dans **Portal** et **Half-Life** proviennent directement de ces observations.

L'objectif n'est pas de demander l'avis des joueurs.

L'objectif est d'observer leur comportement.

---

# 10.18 Étude de cas : Nintendo

Nintendo est réputé pour ses prototypes extrêmement simples.

Avant de produire les graphismes définitifs :

- les déplacements sont validés ;
- les sauts sont ajustés ;
- les sensations sont peaufinées.

Les graphismes arrivent seulement lorsque le gameplay fonctionne déjà parfaitement.

---

# 10.19 Checklist avant un Playtest

Avant de lancer un test :

✓ Les objectifs sont clairement définis.

✓ Une seule hypothèse est testée.

✓ Les commandes fonctionnent.

✓ Les bugs bloquants sont corrigés.

✓ Les données à recueillir sont connues.

✓ Les observateurs prennent des notes.

---

# Exercices

## Exercice 1

Imaginez un prototype de votre jeu.

Listez uniquement les fonctionnalités indispensables.

Supprimez tout le reste.

---

## Exercice 2

Préparez une fiche d'observation.

Ajoutez les colonnes suivantes :

- Temps
- Action du joueur
- Difficulté rencontrée
- Réaction observée
- Solution envisagée

Utilisez cette fiche pendant un Playtest.

---

## Exercice 3

Demandez à une personne de jouer à votre prototype pendant 15 minutes.

Pendant toute la session :

- ne donnez aucune explication ;
- prenez des notes ;
- posez vos questions uniquement à la fin.

Analysez ensuite les résultats et définissez trois améliorations prioritaires.

---

# À retenir

- Le prototype sert à tester une idée, pas à produire un jeu final.
- Un prototype doit répondre à une hypothèse précise.
- Les Playtests permettent d'observer les comportements réels des joueurs.
- Les observations sont souvent plus utiles que les opinions.
- Les données qualitatives et quantitatives se complètent.
- Les retours doivent être priorisés avant d'être intégrés.
- L'itération est le cœur du métier de Game Designer : concevoir, tester, apprendre, améliorer.
